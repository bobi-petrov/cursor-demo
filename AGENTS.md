# TeamFlow — Cursor workshop (session 1)

## Workshop setup

- **Session 1** does **not** use Cursor **Skills** or **subagents**. This repo intentionally has no `.cursor/skills` or `.cursor/agents` directories—do not ask participants to enable them for this exercise.
- **Team A:** use a vague prompt only; optionally clear/disable **User Rules** in Cursor for fairness. In-repo: **no** project rules (empty `.cursor/rules` / omit checkout of Team B assets if you branch).
- **Team B:** keep root **`AGENTS.md`** and the single project rule **`.cursor/rules/workshop-task-validation.mdc`**. In Cursor **Settings → Rules**, set that rule to **Always apply** if you want it without opening `app/api/tasks/route.ts`.

## App context

**TeamFlow** — small task app for teaching: Next.js 14 (App Router), TypeScript strict, Tailwind, Prisma + SQLite, JWT auth (httpOnly cookie, not production-hardened). Task creation lives in **`app/api/tasks/route.ts`**.

## Exercise (canonical task)

Add **input validation** to the **task creation** API: **`POST`** in [`app/api/tasks/route.ts`](app/api/tasks/route.ts).

- **`title`:** required; after trim, non-empty; length **strictly less than 100** characters (max 99).
- **`projectId`:** required; after trim, must match **Prisma cuid-style** ids in this repo (**not** RFC-4122 UUID). Real project IDs look like cuids generated by Prisma (`length` 25, pattern `^[a-z][a-z0-9]{24}$`).
- **Errors:** invalid JSON and validation failures → **`400`** with JSON body shaped as in **`workshop-task-validation.mdc`** (`error` + `issues` array). Keep **`401`** / **`404`** semantics as today after validation passes where applicable.

Prefer a **minimal diff**. No new dependencies unless the facilitator allows. Do not “fix” the dashboard, auth, or unrelated routes unless instructed.

## Definition of done

- `npm run build` passes; TypeScript clean.
- `npm run lint` passes for touched files when the repo already uses it.
- Summarize **what changed** and **why** for reviewers.
- If runtime behavior changes and you add tests, match existing Jest layout; otherwise note tests were out of scope.
- Schema changes (if any): **new** Prisma migration only; never edit past migration SQL; keep README migrate instructions accurate.

When a task **explicitly** requires validation (like this one), **closing that gap for the named route** is intended—ignore any older “preserve missing validation” guidance from generic prompts.
